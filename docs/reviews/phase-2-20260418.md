## Summary

디자인 토큰은 바로 Tailwind와 CSS 변수로 이식 가능한 구조(§12)이며, 23개 컴포넌트가 TypeScript Props·Variants·States로 명시돼 개발 핸드오프가 매끄럽다. 와이어프레임은 Mobile/Desktop 양쪽을 ASCII로 제공하고 Hero/Grid/Tabs 등 주요 상호작용을 구체화했다. Blocker/Major 없음. 몇몇 **수치 불일치**(버튼 sm 크기 vs 터치 타겟), **상태 커버리지 누락**(Galleries 섹션), **포맷 미정의**(dateDisplay) 만 정리하면 핸드오프 품질이 완결된다.

## Strengths

1. **토큰-코드 간 간극 최소화** — CSS Variable + Tailwind extend 스니펫이 §12에 완성돼 있어 Phase 3 부트스트랩 시 카피-페이스트로 충분.
2. **접근성의 기본값화** — Focus ring, reduced-motion, 아이콘 버튼 `aria-label`, Esc 닫기, `aria-current` 등 a11y 규약이 §24에 표로 정리돼 컴포넌트 단위 누락 방지.
3. **상태 UI의 구조화** — Loading/Empty/Error/Toast를 별도 컴포넌트로 분리(§13~16)하고 `offline`/`404` 전용 화면도 와이어프레임에 존재.
4. **PWA 비주얼 구체화** — 아이콘 세트 4종, 스플래시 색, `theme_color`, iOS 수동 가이드 Modal 플로우까지 §11·§21에 정의.
5. **결정 반영 확실** — Phase 1 사용자 결정(미니멀 모노크롬, `photography`, Gallery 상세 Must 격상, 런던 Borough 필터)이 관련 문서에 모두 반영됨.

### 🟡 Minor

**[m-01] `Button` 사이즈 `sm`(32px)이 자체 a11y 규약(44×44)과 충돌**
- 위치: `docs/component-spec.md` §1 Sizes / §24 공통 접근성 규약
- 내용: §24에 "터치 타겟 ≥ 44×44" 보장 명시, 그러나 Button `sm=32`, FilterChip `height 32`. `IconButton`은 "패딩으로 확장" 주석이 있지만 `Button sm`에는 없음.
- 근거: 공통 a11y 표와 개별 컴포넌트 Size 표가 상호 모순.
- 권고 방향: (a) 모바일 컨텍스트에서 sm 사용 금지 규약을 명시, 또는 (b) sm의 클릭 영역을 패딩으로 44로 확장하고 시각 높이만 32로 보이게 처리.
- 영향: 모바일 터치 정확도 ↓, a11y 일관성 훼손. Phase 3 구현 전 규약 확정 필요.

**[m-02] `/galleries` 및 `/galleries/[slug]` 상태 UI 명시 누락**
- 위치: `docs/wireframes.md` §4
- 내용: Home/List/Detail/Map/Search는 Loading·Empty·Error 상태가 명시적 / Galleries 섹션은 없음. Gate 1에서 F-08이 Must로 승격됐으므로 같은 수준의 상태 정의 필요.
- 권고 방향: `/galleries`에 "No galleries found"(현재 시드로는 발생 X), 로딩 스켈레톤 / `/galleries/[slug]`에 "Gallery not found" 404 경로, 네트워크 에러 ErrorState.
- 영향: Developer가 임의 해석하여 상태별 UI 편차 발생 가능.

**[m-03] 날짜 표기 포맷 미정의 — 와이어프레임 내 혼용**
- 위치: `docs/wireframes.md` §1/§2(`~28 Apr`), §3(`24 Sep 2026 – 22 Feb 2027`), §0(`~15 Sep`)
- 내용: 카드 메타는 축약형, 상세는 풀포맷이라는 규칙은 합리적이지만 **문서에 명시 없음**. `Intl.DateTimeFormat` 로케일(ko-KR? en-GB?)도 미정.
- 근거: SSR/CSR 날짜 포맷 불일치는 hydration mismatch 원인 중 대표적.
- 권고 방향: `lib/utils.ts`에 `formatDateShort` / `formatDateRange` 두 함수와 `en-GB` 로케일 고정을 문서화.
- 영향: Phase 3에서 hydration 경고 발생 및 카드/상세 간 포맷 편차.

**[m-04] Gallery 로고 fallback 불일치**
- 위치: `docs/design-system.md` §9.1 ("없을 시 40x40 PNG") vs `docs/component-spec.md` §5 ("없으면 이니셜")
- 내용: 둘 중 어느 쪽이 규칙인지 불명확. 시드에 `logoUrl` 없는 미술관이 다수일 가능성.
- 권고 방향: "SVG 우선 → 없을 시 **이니셜 아바타**(첫 2글자 대문자)"로 단일화하고 디자인 토큰(배경색, 텍스트색) 명시.

**[m-05] Primary CTA 중복/부재 — 2개 화면**
- 위치: `docs/wireframes.md` §4 `/galleries/[slug]` 의 `[Website ↗]`, §9 `/not-found`의 두 버튼
- 내용: Gallery 상세의 "Website" 버튼 variant 미기입 (Primary? Secondary?). 404는 "Back to home"과 "Search exhibitions"가 동일 비중 → Primary 식별 불가.
- 근거: 공통 원칙 "화면당 Primary CTA 1개" (reviewer.md Phase 2 체크리스트).
- 권고 방향: Gallery 상세는 "Website"를 Secondary, (전시가 있으면) 대표 전시 카드를 강조로. 404는 "Back to home"을 Primary.

**[m-06] 다수 컴포넌트의 States 표 누락**
- 위치: `docs/component-spec.md` §7 FilterSheet / §8 Tabs / §9 Dialog / §10 BottomSheet / §11 Map·Marker / §16 Toast / §20 Lightbox
- 내용: Variants와 Props는 있으나 **상호작용 상태**(opening/open/closing, selected, loading 등) 표가 일부만 존재. §8 Tabs는 active 묘사가 산문에만.
- 권고 방향: 각 컴포넌트에 "States" 하위 섹션을 최소 default/hover/active/focus/disabled 5열 표로 통일.
- 영향: Phase 3에서 시각적 일관성 검증이 어려워짐 (리뷰어가 기준 문서 인용 불가).

### ⚪ Nit

**[n-01] `ExhibitionCard` `horizontal` variant와 "Other at this gallery" 매핑 불명확**
- `§3` 와이어프레임의 "horizontal scroll, 2 visible"이 `horizontal` variant인지 `default`를 가로 스크롤에 넣은 것인지 모호.

**[n-02] `/exhibitions/[slug]` 로딩 상태 명시 없음**
- SSG라 런타임 로딩 스켈레톤 필요 없음은 맞지만, 이미지 지연/네트워크 에러 ErrorState는 명시 권장.

**[n-03] 데이터 바인딩 기술의 불균일**
- Home(`§1`)에만 "데이터 바인딩" 섹션. 다른 화면도 어떤 필드가 어디 매핑되는지 1~2줄 추가하면 Developer 해석 편차 ↓.

**[n-04] `/` h1 선택 (전시 제목) vs 사이트 h1**
- SEO 관점에서 검색엔진이 Home의 주제를 "Turner Prize 2026"으로 오해할 수 있음. 시각적 강조는 Hero Title로 두고, 논리적 h1은 `sr-only`로 "GalleryUK — Currently on view in the UK"를 권장.
