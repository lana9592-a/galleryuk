### 🟡 Minor

**[m-01] Props 타입 export가 `Button`·`IconButton`에만 있음**
- 위치: `components/ui/{Badge,Container,EmptyState,ErrorState,Skeleton}.tsx`, `components/{gallery,exhibition,layout}/*.tsx`
- 내용: `agents/developer.md §8`은 "컴포넌트는 Props 타입 export (스토리/테스트 재사용)"을 규약으로 명시. 실제로는 대부분 컴포넌트가 inline anonymous 타입(`function X({…}: { … })`)을 사용해 재사용 불가.
- 근거: `components/exhibition/ExhibitionCard.tsx:22-32`, `components/gallery/GalleryCard.tsx:8-17`, `Badge.tsx:9-14` 등.
- 영향: Phase 5에서 Vitest/Playwright 픽스처 작성 시 Props 타입을 손으로 다시 선언해야 함. 스토리북 도입 시에도 마찰.
- 권고 방향: 각 컴포넌트에 `export type <Name>Props = { … }` 선언 추가 (기존 `ButtonProps` 패턴 답습).

**[m-02] 라우트 그룹 `app/(site)/` 미적용**
- 위치: `app/layout.tsx` (루트 레이아웃) vs `agents/developer.md §2` (폴더 구조).
- 내용: 규약 문서는 `app/(site)/layout.tsx`로 UI 레이아웃을 그룹화하고 `app/api/*`, `app/offline/`, `app/manifest.webmanifest` 등을 그룹 밖에 두기를 권한다. 현재는 모든 라우트가 루트 layout을 공유하고 있어, Phase 4에서 오프라인 페이지를 Header/BottomNav 없이 렌더하려 할 때 레이아웃 오버라이드 재설계가 필요.
- 근거: 현재 `/offline`도 `Header`·`BottomNav`가 따라붙음 (`app/layout.tsx:42-47`).
- 영향: Phase 4 PWA 오프라인 fallback 품질과 충돌 가능.
- 권고 방향: (a) `app/(site)/layout.tsx`로 이전 후 `app/offline/layout.tsx`를 독립 선언, 또는 (b) 현재 구조 유지하되 `/offline`을 Service Worker가 가리키는 단일 HTML로 별도 구현한다는 결정을 `CLAUDE.md`/`README.md`에 기록.

**[m-03] `ExhibitionCard` 접근성 레이블이 upcoming 상태에서 부정확**
- 위치: `components/exhibition/ExhibitionCard.tsx:34` — `\`${title}, ${gallery}, ends ${endDate}\``
- 내용: Home/Galleries 상세의 "Coming soon" 섹션에서도 동일 카드가 렌더되는데, 아직 시작 전인 전시를 스크린리더가 "ends 12 Sep"로만 안내 → 오해 유발. `getExhibitionStatus()`를 이용해 분기 필요.
- 근거: `app/page.tsx:77-85`, `app/galleries/[slug]/page.tsx:125-133`에서 upcoming 목록에 동일 카드 사용.
- 영향: WCAG 2.1 SC 1.3.1 / SC 2.4.4 링크 목적 명확성 경계 (AA 실패까진 아니나 품질 저하).
- 권고 방향: upcoming인 경우 "opens 3 Oct", ended인 경우 "ended 18 Apr" 등으로 동적 전환.

**[m-04] `/search`가 서버 렌더링으로 구현됨 (developer.md §5는 CSR)**
- 위치: `app/search/page.tsx` + `agents/developer.md §5` 라우팅 전략 테이블
- 내용: 규약은 "입력 반응형" 전제로 CSR을 지정하나, 현재는 GET form 제출 기반 SSR. 타이핑 중 즉시 결과 업데이트(debounce)·최근 검색어·키보드 네비는 불가. Phase 4에서 재작성 비용 발생.
- 영향: 사용자 대기 인식 시간(typed → submit 왕복) 증가. MVP 수용 가능하지만 규약 불일치.
- 권고 방향: 현 단계에서는 허용하되, Phase 4 목표에 "검색 CSR 전환 + `/api/exhibitions?q=`" 항목을 `CLAUDE.md §Phase 4`에 추가.

**[m-05] `/map` 페이지가 Phase 4 의존 기능을 암시하는 플레이스홀더**
- 위치: `app/map/page.tsx:18-30` — "Interactive map with gallery markers is coming in Phase 4."
- 내용: 사용자 카피에 "Phase 4"라는 내부 용어가 노출됨. 공개 프리뷰 URL을 누구에게든 공유하면 문맥 없는 단어로 읽힘.
- 영향: 배너 스크린샷/OG 공유 시 프로페셔널리즘 손상.
- 권고 방향: "Interactive map is coming soon" 같은 외부용 카피로 바꾸고, 개발자 메모는 주석으로 이동.

**[m-06] `CATEGORY_LABEL` fallback이 타입-dead 경로**
- 위치: `components/exhibition/ExhibitionCard.tsx:62`, `app/exhibitions/[slug]/page.tsx:85` — `CATEGORY_LABEL[e.category] ?? e.category`
- 내용: `e.category`는 Zod enum으로 좁혀진 5개 리터럴, `CATEGORY_LABEL`은 5 키 모두 정의됨 → `??` 분기 도달 불가. 방어 코드가 오히려 규약 "매직 상수 없음"(§8) 취지와 어긋나는 노이즈.
- 영향: 실질 없음, 가독성/의도 전달만 문제.
- 권고 방향: `CATEGORY_LABEL`을 `Record<Category, string>`으로 재선언하고 `??` 제거.

### ⚪ Nit

**[n-01] Link `href`에 `as never` cast 남발**
- 위치: 14건 — `app/exhibitions/[slug]/page.tsx:154`, `components/exhibition/ExhibitionCard.tsx:38`, `components/gallery/GalleryCard.tsx:21` 등
- `typedRoutes` 하에서 템플릿 리터럴이 `Route` 유니온에 매치되지 않아 우회. `as Route`(`import type { Route } from 'next'`)가 의도를 명확히 표현.

**[n-02] `formatDateShort` 반환이 "28 Apr." (이중 공백 없이 점 없음)과 "Apr 28"의 기대 혼재 가능성**
- `en-GB` + `day: '2-digit', month: 'short'` → "28 Apr" 로 고정되나 스펙(§12.5) 예시에는 "Apr 28"도 등장. 실제 구현이 `en-GB`로 귀결됐으니 스펙 쪽 Nit.

**[n-03] `Header.tsx`의 desktop 검색 트리거가 `/search`로 Full Navigation**
- 데스크톱 규약상 Header 검색 입력 인라인을 선호할 수 있으나, 모바일 일관성상 현 구현도 합리적.

**[n-04] 반복되는 "버튼처럼 보이는 Link" 클래스 문자열 4곳**
- `EmptyState.linkAsButton`, `app/not-found.tsx` 2곳, `app/exhibitions/[slug]/page.tsx` 부킹 CTA 등. 향후 `<Button asChild>` 패턴 또는 `buttonClass()` 헬퍼로 중복 제거 권장. Phase 4 이전에 할 필요는 없음.
