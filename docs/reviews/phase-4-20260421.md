## Strengths
1. **API 경계 방어 일관성** — `ExhibitionsQuerySchema.safeParse` 실패 시 400 + issues 반환 (`app/api/exhibitions/route.ts:10-16`), `lib/api.ts:4-13`에서 `status/category/q/limit` 모두 화이트리스트 또는 범위 제한.
2. **캐시 전략 레이어드** — Route Handler는 `revalidate = 300` + `Cache-Control: public, s-maxage=300, stale-while-revalidate=60`, SW는 `/api/*`에 stale-while-revalidate, navigation에는 network-first → `/offline`, 이미지는 cache-first LRU(60).
3. **Graceful degradation** — `NEXT_PUBLIC_GMAPS_KEY` 미설정 시 `/map`이 점선 placeholder + 항상-보이는 gallery 리스트로 단독 기능 유지 (`app/(site)/map/page.tsx:29-55`). API 키 없어도 빌드/런타임 모두 정상.
4. **외부 I/O 방어** — Book tickets `target="_blank" rel="noopener noreferrer"` + `sr-only` "(opens in new tab)" 안내 (`app/(site)/exhibitions/[slug]/page.tsx:115-122`); 외부 이미지 호스트는 `next.config.mjs`에서 10개 whitelist.
5. **SW 생명주기 위생** — 버전별 네임스페이스(`APP_SHELL/RUNTIME/IMAGES/API` × `v1`), `activate`에서 허용 캐시 외 전부 제거 후 `clients.claim()`, `trimCache`로 이미지 LRU 60개 유지.
6. **클라이언트 자원 관리** — SearchClient는 250ms 디바운스 + AbortController로 in-flight 요청 취소 + URL 동기화 `router.replace({ scroll: false })`. LondonMap은 `useEffect` cleanup에서 마커/클러스터러 해제.
7. **데이터 무결성 보존** — `lib/data.ts:34-40` 미지의 `galleryId` 참조 시 빌드타임에 throw. Phase 4에서도 정적 JSON 기반을 유지하면서 새 route 계층만 얹어 기존 불변식 깨뜨리지 않음.
8. **RSC/CSR 경계 준수** — `/search`, `/map`은 얇은 서버 래퍼가 데이터를 pick해서 클라이언트 컴포넌트로 전달. `'use client'`는 실제로 브라우저 API가 필요한 `SearchClient`, `LondonMap`, `ServiceWorkerRegistrar`에만 존재.

### 🟡 Minor

- **[m-01] SW `networkFirstNavigate`가 non-2xx 응답을 캐싱할 수 있음**
  위치: `public/sw.js:75-88`
  근거: `staleWhileRevalidate`는 `res.status === 200`일 때만 `cache.put`하지만 (`sw.js:95`), `networkFirstNavigate`는 응답 상태와 무관하게 `cache.put(req, fresh.clone())` 수행. 404/5xx/리다이렉트 응답도 캐시되어 다음 오프라인 방문 때 오염된 HTML을 재생할 수 있음.
  영향: 오프라인 UX 품질 저하 가능성 (상급 에러 HTML이 캐시된 채 노출).

- **[m-02] 루트 metadata에 `openGraph.images` 부재**
  위치: `app/layout.tsx:20-26`
  근거: PRD §Share & SEO에서 카드 미리보기 품질을 요구. 현재는 `openGraph.images` 미지정으로 기본 파비콘만 노출됨. 전시 상세 페이지(`app/(site)/exhibitions/[slug]/page.tsx:41`)는 이미 hero 이미지를 OG로 제공하지만 나머지 사이트는 미커버.
  영향: 소셜 공유 시 미리보기 이미지가 비어보임.

- **[m-03] PWA 아이콘이 SVG 전용**
  위치: `public/icons/*.svg`, `app/manifest.ts:16-29`
  근거: README 스스로 "older Androids prefer PNG" 명시(`README.md:132-133`). Lighthouse PWA 카테고리의 "Icons of sufficient size" 체크가 경고로 뜰 수 있으며, iOS splash screen 자동 생성이 SVG에서 지원되지 않음.
  영향: 설치 경험 일부 기기에서 다운그레이드. Phase 5 QA에서 계측 필요.

- **[m-04] `typed routes`와 동적 세그먼트 캐스트**
  위치: `components/map/LondonMap.tsx:128`, `app/(site)/map/page.tsx:65`
  근거: `Link href={\`/galleries/${id}\` as never}` 패턴이 `typedRoutes` 타입 시스템의 template-literal 한계 때문에 반복됨. 기존 `app/(site)/...` 페이지에서도 유사 패턴이 있으나 한 지점씩 늘어나고 있어 공통 헬퍼(`galleryHref(id)`)로 정리 권장.
  영향: 코드 위생. 향후 라우트 변경 시 검색 대상이 늘어남.

- **[m-05] SearchClient 초기 렌더의 "빈 결과" 짧은 flash**
  위치: `components/search/SearchClient.tsx:26-28, 51-57`
  근거: `initialQuery`가 truthy면 `results`가 null로 시작해 첫 프레임에서 "Type a keyword…" 안내가 잠깐 보일 수 있음(`debounced`는 이미 `initialQuery`이지만 fetch 시작 전 플레임). 초기 마운트 시 `setLoading(true)`를 선제 세팅하면 해결됨.
  영향: URL에 `?q=...`로 직접 진입 시 체감 UX에 hitch.

- **[m-06] `VERSION = 'v1'` 수동 핀**
  위치: `public/sw.js:8`
  근거: 배포 시마다 SW가 바뀌어도 캐시 이름이 고정되어 오래된 에셋을 재사용할 위험. 빌드 해시(예: `process.env.NEXT_PUBLIC_BUILD_ID` 주입 또는 빌드 시 치환)로 묶어야 "activate → cleanup"이 실제로 새 버전에서 작동.
  영향: 배포 후 사용자가 이전 번들을 끌고 다닐 수 있음. Phase 5 배포 전 반드시 처리.

- **[m-07] `/api/galleries`의 `activeCount`가 캐시 창 내 경계 전시에서 어긋날 수 있음**
  위치: `app/api/galleries/route.ts:14-19`
  근거: `new Date()` 시점 기반으로 계산하나 `s-maxage=300`으로 5분간 재사용 → 시작/종료 경계에 있는 전시가 최대 5분간 잘못된 집계로 노출될 수 있음.
  영향: 낮음 — 본질적으로 정보성 뱃지. v1.0 허용 범위지만 문서에는 기록 필요.

### ⚪ Nit

- **[n-01] Manifest `description`이 `SITE_DESCRIPTION`과 동일** — PWA 설치 UI용 카피는 "한 곳에서 London 전시 일정 확인" 같은 짧은 action-oriented 문장이 더 나음 (`app/manifest.ts:8`).
- **[n-02] LondonMap 닫기 버튼 `✕` 유니코드** — 나머지 UI는 `lucide-react` 아이콘만 사용. 일관성을 위해 `<X aria-hidden />` 권장 (`components/map/LondonMap.tsx:124`).
- **[n-03] LondonMap `zoom={12}` 하드코드** — 마커 수 증가 시 시야 이탈 가능. `map.fitBounds(LatLngBounds)`로 전환 시 갤러리 추가에도 자동 적응 (`components/map/LondonMap.tsx:99`).
- **[n-04] `lib/api.ts:44` 정렬 기준이 `status==='upcoming'`일 때만 분기** — `status==='ended'`의 경우 "가장 최근 종료" 순(내림차순)이 더 자연스러움. 현재는 `endDate` 오름차순으로 "가장 오래된 ended"가 상단.
